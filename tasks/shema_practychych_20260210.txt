Лабораторні  заняття  відбуваються  в  очному  режимі  (в  аудиторії), 
інше  не  регламентується  деканатом  факультету  прикладної 

якщо 
математики та інформатики.  

Cтуденти  виконують  вказані  у  цих  вказівках  завдання,  отримують 
необхідну допомогу від викладача та можливість спілкування з колегами та 
викладачем у форматі семінарів/лабораторних.  

На  початку  лабораторних  занять  студентам  пропонується  виконати 
перше  завдання  суть  якого  полягає  у  виборі  предметної  області, 
функціональних 
і  нефункціональних  вимог  та  розробці  початкової 
архітектури  системи.  Студент  може  запропонувати  свою  предметну 
область,  або  ж  розглянути  одну  із  пропонованих  систем:  наприклад, 
побудову  сервіса  для  розподіленого  пошуку  у  файлах,  чи  система  
векторизації текстів для LLM на базі Apache Kafka.  

Пропоновані  сервіси  будуть  розглядатися  на  лекціях  детально  “з 

нуля”.  

Далі,  cтуденти  можуть  працювати  на  лабораторних  в  одному  із  двох 

форматів, попередньо узгодивши із викладачем лабораторних:  

1.  Стандартний формат.

практичних заняттях. Відвідування практичних занять є 
обов’язковим. Завдання здаються індивідуально на практичних 
заняттях у наперед встановлений термін. Завдання не здані в термін 
можуть бути здані за наявності вільного часу наприкінці семестру із 
максимальною оцінкою, що становить 50% від початкової 

2.  Альтернативний формат. 

Робота над власним проектом чи освоєння певного курсу (udemy,

Етап завдання і основні вимоги: 

Завдання  моделює  реальні  виклики  при  підготовці  даних  для  навчання 
сучасних Large Language Models (LLM). 

Ви працюєте у команді розробки LLM. Ваше завдання — побудувати 
пайплайн, який перетворює потік сирих текстових статей у числові 
вектори, придатні для навчання нейронної мережі. 

Проблема: Класичні підходи (наприклад, sklearn.TfidfVectorizer) 
тримають весь словник у пам'яті однієї машини, що неможливо при 
обробці терабайтів даних. 

Мета: Розробити розподілену систему, яка: 

1.  Формує статичний словник на основі історичних даних (Warm-up 

Phase). 

2.  Обробляє потік нових документів у реальному часі (Production 

Phase). 

3.  Генерує розріджені вектори (Sparse Vectors) на основі метрики TF-

IDF. 

4.  Використовує Apache Kafka як єдиний засіб комунікації та 

збереження стану (State Persistence). Використання зовнішніх БД 
(Postgres/Redis) не рекомендовано. 

Пропозиція Архітектура Системи  
(може бути змінено студентом) 

Система повинна працювати у двох режимах (фазах). Перемикання між 
режимами ініціюється спеціальною керуючою подією. 

Фаза 1: Навчання (Warm-up / Learning) 

1.  Вхід: Історичний дамп даних (bootstrap-dataset). 
2.  Дія: Система рахує глобальну частоту слів. Вектори не генеруються. 

 
 
 
3.  Результат: Побудова словника (Vocabulary) та глобальної 

статистики (IDF Stats). 

Фаза 2: Продакшн (Streaming / Production) 

•  Вхід: Потік нових статей (raw-articles). 
•  Дія: Воркери використовують сформований словник для 

перетворення тексту у вектори. 

•  Результат: Потік векторів у топіку ml-vectors. 

Студент  має  реалізувати  три  компоненти,  які  взаємодіють  через  Kafka 
топіки. Система в цілому реалізує спрощений аналог MapReduce. 

Компонент 1: Stats Reducer (Aggregator) 

1.  Stateful сервіс. Запускається в єдиному екземплярі (або з правильним 

партиціюванням). 

2.  Слухає топік word-counts. 
3.  Агрегує дані в пам'яті (рахує DF для кожного слова та глобальне N). 
4.  Періодично скидає абсолютні значення в топік corpus-stats. 
5.  Спеціальна логіка: При отриманні команди "Finalize Vocabulary" (або

•  Бере Топ-X найпопулярніших слів. 
•  Присвоює їм ID від 0 до X. 
•  Публікує словник у топік vocabulary-map.

Scalable  сервіс.  Запускається  N  екземплярів.  Працює  як  State  Machine  з

2.  Відправляє унікальні слова документа в word-counts (лише 1 раз для

3.  Не генерує повідомлень у ml-vectors. 
4.

2.  Читає raw-articles.
